# Plan: Edgar plugin

> **Status: implemented as a built-in plugin.** Shipped under `src/plugins/edgar/` (frontend-facing META + tool definition + registration) and `server/edgar/` + `server/api/routes/edgar.ts` (host-side client + Express dispatch). Wraps the public SEC EDGAR API as one tool with kind-discriminated dispatch, server-only (no Vue View / Preview), with a self-healing config flow when the SEC-required contact info is missing.

## Why built-in (not runtime)

Initial design routed edgar through the runtime-plugin pipeline (`packages/plugins/edgar-plugin/` + `server/plugins/preset-list.ts`). That hit a structural problem: the host's runtime-plugin loader auto-includes every loaded runtime plugin in every role's tool set regardless of `role.availablePlugins`, so edgar's ~70-line tool description would land in every role's system prompt ‚Äî even roles that have nothing to do with SEC filings.

Built-in plugins ARE gated by `role.availablePlugins` (the standard pattern that accounting / chart / canvas / presentForm follow). Edgar fits the built-in shape cleanly because:

- It has a single MCP-bridge dispatch route (`POST /api/edgar`).
- It has no Vue View ‚Äî but `PluginEntry.viewComponent` and `previewComponent` are both optional, so server-only built-in plugins are fully supported.
- Its tool name (`edgar`) is a normal `TOOL_NAMES` member after codegen, so roles reference it as `TOOL_NAMES.edgar` like every other built-in.

The runtime-plugin path stays appropriate for plugins that the user installs as separate npm packages outside the repo (the `~/mulmoclaude/plugins/plugins.json` workspace ledger). Anything we ship in-tree should be a built-in.

## Tool surface

One tool, name `edgar`, with a `kind` discriminator covering all six EDGAR endpoints:

- `lookup_cik` ‚Äî ticker ‚Üí 10-digit CIK + company name.
- `get_recent_filings` ‚Äî list a company's most recent filings (filterable by `form_types`).
- `get_filing_document` ‚Äî fetch the primary document of a specific filing as raw HTML/text. `max_chars` defaults to 20 000 (vs 50 000 in the standalone MCP) ‚Äî without a View the LLM has to read this verbatim.
- `get_company_facts` ‚Äî every XBRL-tagged fact (large; prefer `get_concept`).
- `get_concept` ‚Äî time series for one XBRL concept across all filings.
- `search_filings` ‚Äî full-text search across the entire EDGAR corpus.

Single tool over six was chosen because `roles.ts` gates by tool name; one entry in `availablePlugins` keeps role configs short and the LLM's tool list compact.

## Config ‚Äî self-healing missing-config flow

Config lives at the literal path:

```
~/mulmoclaude/config/plugins/edgar/config.json
```

Schema:

```json
{ "name": "Full Name", "email": "user@example.com" }
```

The route reads the config first. If absent, it returns (does NOT throw) the `config_required` payload:

```json
{
  "error": "config_required",
  "instructions": "The SEC EDGAR API requires an identifying contact on every request. Please ask the user for their full name and email address, then write a JSON file at the absolute path below with the exact schema below, then retry the original tool call. Do not proceed without asking the user ‚Äî never invent a name or email.",
  "path": "/Users/.../mulmoclaude/config/plugins/edgar/config.json",
  "schema": { "name": "<user's full name>", "email": "<user's email address>" }
}
```

The LLM treats this as data, asks the user for name+email, uses its built-in `Write` tool to create the file, and retries. No throw, no UX dead-end, no in-app Configure form.

## Host wiring

- `META` is auto-discovered by `scripts/codegen-plugin-barrels.ts` because the directory matches the `src/plugins/<name>/` convention. Running `yarn plugins:codegen` regenerates the three barrel files.
- The route must be mounted manually ‚Äî added `import edgarRoutes from "./api/routes/edgar.js"` and `app.use(edgarRoutes)` next to the other route mounts in `server/index.ts`.
- HTTP allowlist is enforced inside `client.ts` (`ALLOWED_HOSTS = {"www.sec.gov", "data.sec.gov", "efts.sec.gov"}`) ‚Äî refuses to hit any other host even if the URL string is constructed elsewhere.

## Role wiring

No default role lists `TOOL_NAMES.edgar` in this PR ‚Äî it's opt-in per role. Users add it to a custom role:

```ts
{
  // ...
  availablePlugins: [TOOL_NAMES.presentForm, TOOL_NAMES.edgar, ...],
}
```

Because edgar is a built-in plugin, its tool name is a real `TOOL_NAMES` member after codegen, so the `TOOL_NAMES.edgar` literal is type-safe ‚Äî typos surface at compile time.

## Out of scope (explicitly)

- **No View / Preview.** Pure server-side tool. Adding a canvas surface later is a separate PR.
- **No streaming.** Each dispatch is a single round-trip; large responses come back whole. The LLM applies its own truncation when summarising.
- **No XBRL post-processing.** `get_company_facts` and `get_concept` return SEC's JSON shape unchanged; the LLM interprets it.
- **No retry logic** beyond what `fetch` provides. EDGAR rate-limits at 10 req/sec and we throttle to 9; a 429 still bubbles up as an error.
